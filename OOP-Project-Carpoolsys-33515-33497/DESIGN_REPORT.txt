1. OVERVIEW
================================================================================

IBA has two campuses across Karachi and many students come from the same
neighborhoods with no way to coordinate rides, leading to solo trips and
wasted fuel. The IBA Carpool Facilitation System lets student drivers post
rides and passengers search and book them. It enforces a campus rule (every
ride must involve a campus), respects gender preferences, and persists data
across sessions.

Built in C++17 with a Raylib GUI at 1920x1080. Data is stored in CSV files.
Covers all required OOP concepts: inheritance, polymorphism, encapsulation,
operator overloading, copy constructors, STL, recursion, and sorting.

The system has three layers: a data layer (four CSV files read at startup,
written after every change), a logic layer (CarpoolSystem owns all in-memory
maps and enforces all business rules), and a presentation layer (gui.cpp,
compiled only with USE_GUI, driven by an enum class Screen state machine).
Both user roles use the same Student class differentiated by an isDriver flag.


================================================================================
2. DESIGN AND OOP
================================================================================

Class Hierarchy
  Student stores ID, name, phone, gender, role flag, and a ride counter.
  Vehicle is an abstract base with Car and Bike as subclasses. It holds shared
  fields (vehicleId, ownerId, make, model, plate, capacity) and one pure
  virtual method getType() -- Car returns "Car", Bike returns "Bike", no casts
  needed anywhere. Ride represents a posted journey: owns its passenger ID
  list, seat count, date/time, areas, gender preference, and status. It stores
  IDs rather than pointers to avoid ownership issues and keep CSV serializing
  simple. Notification is a recipient ID and message string, session-only.
  CarpoolSystem owns four std::map collections and all ID counters; every
  state-changing operation is a method here.

Encapsulation
  All data members are private. Getters are const-qualified so the GUI can use
  const pointers safely. Setters exist only where a field legitimately changes
  after construction (profile editing for Student/Vehicle; file-load helpers
  for Ride). rawAddPassenger bypasses normal checks and just appends an ID --
  intentional because when loading from CSV, seats and status are restored
  separately and double-counting must be avoided.

Polymorphism
  vehicles stores Vehicle* heap pointers. Calling v->getType() dispatches
  through the vtable to the correct subclass -- no switch statements in the
  caller. The same virtual dispatch works inside operator<<(ostream, Vehicle).
  CarpoolSystem's destructor deletes every pointer in the map.

Operator Overloading
  Student: operator== and operator!= compare by ID; friend operator<< streams
  a one-line summary. Ride: operator< compares chronologically by date then
  time (YYYY-MM-DD and HH:MM both sort lexicographically), making Ride work
  directly with std::sort; operator== compares by ride ID; friend operator<<
  streams route, status, and seats. Vehicle: friend operator<< streams type,
  make, model, plate, and capacity with correct type label via virtual dispatch.
  All operator<< are friend functions because the left operand is ostream&.

Copy Constructor
  Student has an explicit copy constructor that member-initializes all fields.
  The compiler-generated version would behave identically, but the explicit
  definition satisfies the course requirement and puts deep-copy logic in the
  right place if a pointer member is ever added. Students are stored by value
  in std::map so the copy constructor is called on every insert.


================================================================================
3. IMPLEMENTATION
================================================================================

Data Structures and Algorithms
  std::map is the primary container for all four collections. O(log n) keyed
  lookup is the most common operation, and ordered iteration keeps CSV output
  stable across runs. Passenger lists inside Ride use std::vector<string>;
  the list is always small (capped by vehicle capacity) so linear scan and
  index-based erase are both fine.

  searchRides filters by matching areas, OPEN/FULL status, and departure time
  within 30 minutes of the requested time. Results are sorted by std::sort
  with a lambda (closest time first), with Ride::operator< as a chronological
  tie-breaker.

  The Apriori algorithm identifies students who frequently travel together.
  Each completed ride is a transaction (sorted list of all participants).
  Step 1 counts singletons appearing in at least minSupport rides. Steps 2
  and 3 find frequent pairs and triples using the same recursive genItemsets
  function rather than two separate loops. It uses backtracking: build a
  current vector by adding one candidate at a time (always from a forward
  index to avoid duplicates), count matching transactions when target size k
  is reached, then pop and try the next candidate. Passing different k values
  handles pairs and triples with the same code.

Persistence
  CSV was chosen because it is human-readable and needs no external library.
  Four files match the four entity types. Every mutation writes the relevant
  CSV immediately so data on disk always matches in-memory state, even after
  a crash. The loader handles both old and new field counts in students.csv,
  and detects whether a date field is present in rides.csv (old rides get
  today's date). Passengers are stored as a semicolon-separated list in the
  last rides.csv field, keeping one row per ride.

GUI
  The GUI is a single private class (CarpoolGUI) in gui.cpp; the only public
  interface is runGUI(CarpoolSystem&). Screen transitions go through
  setScreen(), which clears the error message and runs checkAutoComplete().
  Small static widget helpers (btn, panel, textInput, areaGrid, etc.) handle
  hover detection, colour logic, and click detection, keeping call sites
  compact. Form state is stored in member variables prefixed by reg_, cr_,
  sr_, ep_ so values survive frame resets. Lists use integer scroll offsets
  clamped each frame; only the visible slice is drawn.

  Colour system: deep maroon background (C_BG), white content panels
  (C_PANEL), IBA crimson (C_ACCENT) for action buttons and selected states,
  green (C_CAMPUS) for campus areas in the area grid.

Validation
  phoneTextInput rejects non-digits and caps at 11 characters. The area grid
  prevents identical pickup and dropoff. Date buttons are pre-computed so no
  invalid date string is possible. Registration and edit-profile handlers use
  try/catch around runtime_error -- the caught message is stored in errMsg and
  shown in red below the form. createRide, bookRide, and cancelRide each guard
  independently of the GUI so business rules hold in any build.


================================================================================
4. CHALLENGES AND CONCLUSION
================================================================================

Challenges
  The OOP structure adds some verbosity over a single-file approach but is
  natural rather than forced -- adding a new vehicle type only needs a new
  Vehicle subclass. CSV keeps the project dependency-free beyond Raylib but
  rewrites full files on every save and has no query language; fine at this
  scale, a production system would use a proper database. Notifications are
  not persisted -- a deliberate simplification that would need a fifth file
  or dedicated table to fix. win_icon.cpp uses the Windows API so the project
  only builds on Windows; cross-platform support would need conditional
  compilation. The GUI is hardcoded at 1920x1080; content may clip on smaller
  monitors.

Conclusion
  The system applies OOP principles practically. Inheritance removes type-switch
  logic; virtual dispatch means a new vehicle type is just a new subclass;
  encapsulation keeps internals private; operator overloading makes classes work
  naturally with STL and streams; the explicit copy constructor makes semantics
  auditable; the Apriori backtracking recursion replaces two separate loops with
  one general function; exception handling keeps error reporting out of the main
  flow. The result is a usable, data-safe system that could be extended with a
  database, new vehicle types, or a mobile interface without rewriting its core.
